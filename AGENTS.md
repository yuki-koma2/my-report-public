# AGENTS.md

このリポジトリで作業するエージェント向けの指示です。

## 回答言語

- ユーザーへの回答は日本語で行う。

## リポジトリの目的

- このリポジトリは、公開情報からわかるレポートをまとめて一般公開するためのもの。
- 公開には GitHub Pages を使う。
- ログインや会員機能は不要。
- 実装可能であれば SPA として構築する。

## 実装方針

- GitHub Pages で配信できる静的サイトとして成立する構成を優先する。
- React / Vite / Tailwind CSS / Motion for React を使う。
- クライアントサイドのみで完結できる設計にする。
- 認証、サーバーサイド DB、秘密情報を必要とする構成は避ける。
- 公開情報、出典、確認日を読者が追える UI を重視する。
- 既存の構成や規約が追加された後は、それに合わせて変更する。

## ディレクトリ方針

- `src/` はビルド前の編集対象。
- `dist/` はビルド後の公開対象。GitHub Pages で配信する想定。
- `vite.config.js` は Vite / React / Tailwind CSS の設定。
- `test/` は自動テスト。
- 手作業で `dist/` だけを編集せず、原則として `src/` を変更して `npm run build` で再生成する。

## TDD 方針

t-wada 式 TDD を基本に進める。

1. まず失敗するテストを書く。
2. テストを通す最小限の実装を書く。
3. テストが通る状態を保ってリファクタリングする。

機能追加やバグ修正では、可能な限り先にテストで期待する振る舞いを固定する。テストなしで実装する場合は、その理由を明示する。

## レポート内容の扱い

- 公開情報から確認できる事実だけを扱う。
- 推測、未確認情報、非公開情報を事実として書かない。
- 出典 URL、公開日、確認日をできるだけ記録する。
- 引用は必要最小限にし、長文転載を避ける。
- 古くなりやすい情報は、確認時点がわかるようにする。

## 作業時の注意

- 変更前に現在のファイル構成と差分を確認する。
- ユーザーや他エージェントの未コミット変更を勝手に戻さない。
- 実装後は、利用可能なテスト、lint、build を実行して確認する。
- GitHub Pages の公開設定や CI を変更する場合は、変更理由と確認方法を明確にする。

## デプロイ方針

- GitHub Pages は GitHub Actions 経由で公開する。
- `main` への push で test/build/deploy/tag を実行する。
- pull request では test/build まで実行し、デプロイとタグ作成は行わない。
- デプロイ成功後のタグは `deploy/pages/YYYYMMDD-HHMMSS-<run-number>-<short-sha>` 形式にする。

## スキル方針

- 医療・介護領域の週次調査レポートを作成する場合は `.agents/skills/healthcare-care-weekly-report/` を参照する。
- この skill の出力は、将来サイトへ毎週追加するレポートページの元データとして扱う。
