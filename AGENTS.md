# AGENTS.md

このリポジトリで作業するエージェント向けの指示です。

## 回答言語

- ユーザーへの回答は日本語で行う。

## リポジトリの目的

- このリポジトリは、公開情報からわかるレポートをまとめて一般公開するためのもの。
- 公開には GitHub Pages を使う。
- ログインや会員機能は不要。
- 実装可能であれば SPA として構築する。

## 実装方針

- GitHub Pages で配信できる静的サイトとして成立する構成を優先する。
- React / Vite / Tailwind CSS / Motion for React を使う。
- クライアントサイドのみで完結できる設計にする。
- 認証、サーバーサイド DB、秘密情報を必要とする構成は避ける。
- 公開情報、出典、確認日を読者が追える UI を重視する。
- 既存の構成や規約が追加された後は、それに合わせて変更する。

## Google Analytics 方針

- Google Analytics を入れる場合は GA4 を前提にし、詳細な設計・運用ルールは `docs/google-analytics.md` に従う。
- 測定 ID は `VITE_GA_MEASUREMENT_ID` などの Vite 公開環境変数で渡し、コード内に固定値として埋め込まない。
- 未設定の場合は Analytics のスクリプトを読み込まず、ローカル開発や pull request で不要な計測を発生させない。
- イベント計測を追加する場合は、読者の個人情報、検索語句、問い合わせ内容など、個人や機密に結びつく可能性がある値を送らない。
- Google Analytics の設定や計測対象を変更した場合は、変更理由と確認方法を PR またはコミット本文に残す。

## ディレクトリ方針

- `src/` はビルド前の編集対象。
- `src/app/reportData/` は1ファイル1レポートの本文データ置き場。
- `src/app/tagDefinitions.js` はタグ定義の編集対象。
- `dist/` はビルド後の公開対象。GitHub Actions で生成し、Git管理しない。
- `docs/` は設計、運用ルール、調査方針など、実装に紐づく長めのドキュメントを置く場所。
- `vite.config.js` は Vite / React / Tailwind CSS の設定。
- `test/` は自動テスト。
- 手作業で `dist/` だけを編集せず、原則として `src/` を変更して `npm run build` で生成結果を確認する。

## TDD 方針

t-wada 式 TDD を基本に進める。

1. まず失敗するテストを書く。
2. テストを通す最小限の実装を書く。
3. テストが通る状態を保ってリファクタリングする。

機能追加やバグ修正では、可能な限り先にテストで期待する振る舞いを固定する。テストなしで実装する場合は、その理由を明示する。

## レポート内容の扱い

- 公開情報から確認できる事実だけを扱う。
- 推測、未確認情報、非公開情報を事実として書かない。
- 出典 URL、公開日、確認日をできるだけ記録する。
- 新規レポートの `sources` には、少なくとも `checkedAt` を持たせ、分かる範囲で `publishedAt` も記録する。
- `sources` の `type` は実態に合わせる。一次情報、二次情報、研究論文、配信元リンク、フィードを混同しない。
- Product Hunt、TechFeed、RSS などの配信元フィードを一次情報として扱わない。本文で配信元リンクと分類したものは、出典カードでも同じ種別で表示できるようにする。
- 変化し続けるRSS URLだけを根拠にせず、後から検証が必要な論点には個別記事 URL を出典として残す。個別 URL がない場合は、本文を確認済み出典で説明できる範囲に絞る。
- 出典URLはタイトルと対応する実在ページを確認し、読者がリンク先で根拠を追える状態にする。
- 要点ダッシュボードの件数や期限は、`topicCards` など本文データの実数と必ず一致させる。
- 複数の統計、通知、公募を1つのカードにまとめる場合は、それぞれの公開日・更新日が同一に見えないように分けるか、本文とカードで遡及参照・対象期間外参照を明示する。
- 引用は必要最小限にし、長文転載を避ける。
- 古くなりやすい情報は、確認時点がわかるようにする。

## レポートページの表現方針

- 記事詳細ページは、長い文字列の羅列で終わらせない。
- 冒頭に、その週または記事全体の判断ポイント、要約、重要な期限、注意点を置く。
- 重要度、対応優先度、関連度、期限、一次情報か二次情報かなど、読者が判断に使える指標をカードやチップで可視化する。
- 本文は、トピックカード、アクションカード、メトリクス、引用元カード、タグ、関連主体、期限表示などに分解し、スキャンしやすくする。
- 出典は単なる URL 文字列ではなく、出典名、種別、公開日または確認日、リンクを読める形で表示する。
- 「今週確認できた重要な新規情報なし」のような空情報も、該当テーマ、確認範囲、継続ウォッチ理由が分かる形で表現する。
- `ReportSection` などで特別扱いされるセクションに `section.items` を入れる場合は、詳細ページに実際に描画されることを確認する。描画されない場合は `topicCards` など表示対象の構造へ移す。
- 日本語本文では表記を揃え、文の区切りに半角ピリオドを混ぜない。特に `一次情報。影響を受ける主体` のような定型文は全角句点で統一する。
- UI を追加する場合は、アクセシブルな見出し、リンク、テキストを保ち、テストで主要な表示要素を固定する。

## テスト方針

- `report.sections.find(...)` や `report.sources.find(...)` の結果を参照する場合は、存在確認を先に置くか、オプショナルチェイニングとヌル合体演算子で安全に扱う。対象が見つからない場合に `TypeError` ではなく、期待値の失敗として読めるテストにする。
- セクション本文をカード表示と併せて表示する変更では、同じ文言が複数箇所に出ることがあるため、意図した重複表示では `getAllByText` などを使ってテストを安定させる。

## 作業時の注意

- ローカルで作業するときは、原則として専用の git worktree を作成して作業する。
- `main` 直下で直接作業するのは、初回セットアップやユーザーが明示した軽微な作業など、理由がある場合に限る。
- 変更前に現在のファイル構成と差分を確認する。
- ユーザーや他エージェントの未コミット変更を勝手に戻さない。
- 既存レポートのテストを新規レポートのテストで置き換えない。新しいレポートを追加する場合は、既存レポートの詳細ページ、一覧、タグ、出典表示の回帰テストを保持したまま個別ケースを追加する。
- `reports[0]` など並び順だけに依存する検証を変更する場合は、既存レポートも `id` や `title` で明示的に検証し、レポートが失われていないことを確認する。
- コンフリクト解消後は競合マーカー検索、`npm test`、`npm run build` を実行し、レポート分割やタグ定義の変更で既存ケースが落ちていないことを確認する。
- 実装後は、利用可能なテスト、lint、build を実行して確認する。
- GitHub Pages の公開設定や CI を変更する場合は、変更理由と確認方法を明確にする。

## デプロイ方針

- GitHub Pages は GitHub Actions 経由で公開する。
- `main` への push で test/build/deploy/tag を実行する。
- pull request では test/build まで実行し、デプロイとタグ作成は行わない。
- `dist/` はPR差分に含めず、Actions の build artifact として扱う。
- デプロイ成功後のタグは `deploy/pages/YYYYMMDD-HHMMSS-<run-number>-<short-sha>` 形式にする。

## スキル方針

- 医療・介護領域の週次調査レポートを作成する場合は `.agents/skills/healthcare-care-weekly-report/` を参照する。
- この skill の出力は、将来サイトへ毎週追加するレポートページの元データとして扱う。
